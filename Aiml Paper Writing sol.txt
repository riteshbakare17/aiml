•	Visit 0 → visited={0}, rec_stack={0}, reachable={0}

•	Visit 1 → visited={0,1}, rec_stack={0,1}, reachable={0,1}

•	Visit 3 → visited={0,1,2,3}, rec_stack={0,1,2,3}, reachable={0,1,2,3}

•	1 is already in rec_stack → BACK EDGE DETECTED
•	Return True (cycle found)

•	Remove 3 from rec_stack → rec_stack={0,1,2}
•	Return True to caller
•	Remove 2 → rec_stack={0,1}
•	Return True
•	Remove 1 → rec_stack={0}
•	Return True
•	Remove 0 → rec_stack={}
•	Final result: reachable = {0,1,2,3}, has_cycle = True
✅ Cycle successfully detected!
💡 Key Insight: In DFS, if you encounter a node that’s already in the recursion stack, it means there’s a back edge → cycle!











Q.3 Find Least-Cost Path in Weighted Graph using Uniform Cost Search
Solution:
Let’s trace UCS from 0 → 4 manually.
We use a priority queue (min-heap) ordered by cumulative cost.
Step 0: Initialize
•	PQ: [(0, 0, [0])]
•	Visited: {}
________________________________________
Step 1: Pop (0, 0, [0])
•	Expand node 0 → neighbors: 1(cost 4), 2(cost 1)
•	Push: (4, 1, [0,1]), (1, 2, [0,2])
•	PQ: [(1,2,[0,2]), (4,1,[0,1])]
•	Visited: {0}
________________________________________
Step 2: Pop (1, 2, [0,2])
•	Expand node 2 → neighbors: 0(visited), 1(cost 2), 3(cost 5)
•	Push: (1+2=3, 1, [0,2,1]), (1+5=6, 3, [0,2,3])
•	PQ: [(3,1,[0,2,1]), (4,1,[0,1]), (6,3,[0,2,3])]
•	Visited: {0,2}
________________________________________
Step 3: Pop (3, 1, [0,2,1])
•	Expand node 1 → neighbors: 0(visited), 2(visited), 3(cost 1)
•	Push: (3+1=4, 3, [0,2,1,3])
•	PQ: [(4,3,[0,2,1,3]), (4,1,[0,1]), (6,3,[0,2,3])]
•	Visited: {0,2,1}
________________________________________
Step 4: Pop (4, 3, [0,2,1,3])
•	Expand node 3 → neighbors: 1(visited), 2(visited), 4(cost 3)
•	Push: (4+3=7, 4, [0,2,1,3,4])
•	PQ: [(4,1,[0,1]), (6,3,[0,2,3]), (7,4,[0,2,1,3,4])]
•	Visited: {0,2,1,3}
________________________________________
Step 5: Pop (4,1,[0,1]) → but 1 already visited → skip
Step 6: Pop (6,3,[0,2,3]) → 3 already visited → skip
Step 7: Pop (7,4,[0,2,1,3,4]) → GOAL NODE!
•	Return cost = 7, path = [0,2,1,3,4]
✅ Done! Least cost path found.
💡 Why didn’t we take 0→1→3→4 (cost 8)? Because UCS always expands the cheapest path first — and 0→2→1→3 (cost 4) got expanded before 0→1→3 (cost 5).

Q.6 Implement A* Search to Find Optimal Path Using Heuristic
Solution:
Let’s trace A* from A to G using Euclidean heuristic.
We maintain:
•	g(n) = actual cost from start
•	h(n) = estimated cost to goal (Euclidean distance)
•	f(n) = g(n) + h(n)
Step 0: Initialize
•	Start at A: g=0, h≈8.06 (√(7²+0²)), f≈8.06
•	Open Set: [(8.06, 0, A, [A])]
________________________________________
Step 1: Expand A (lowest f)
•	Neighbors: B(g=3), C(g=5)
•	B: h(B→G)=√((7-2)²+(0-1)²)=√26≈5.10 → f=3+5.10=8.10
•	C: h(C→G)=√((7-1)²+(0-3)²)=√45≈6.71 → f=5+6.71=11.71
•	Open Set: [(8.10, 3, B, [A,B]), (11.71, 5, C, [A,C])]
________________________________________
Step 2: Expand B (f=8.10)
•	Neighbors: A(visited), D, E
•	D: g=3+4=7, h(D→G)=√((7-3)²+(0-4)²)=√32≈5.66 → f=12.66
•	E: g=3+6=9, h(E→G)=√((7-5)²+(0-2)²)=√8≈2.83 → f=11.83
•	Open Set: [(11.71,C), (11.83,E), (12.66,D)]
________________________________________
Step 3: Expand C (f=11.71) → but let’s see if better path through B exists later? Actually, next lowest is E!
Wait — correction: After expanding B, we have:
Open Set: [C(11.71), E(11.83), D(12.66)] → so expand C next.
But after expanding C → D (g=5+2=7), which is same cost as B→D (g=7). So D now has g=7 via C.
Then we expand E (f=11.83):
•	E neighbors: B(visited), F, G
•	G: g=9+5=14, h=0 → f=14 → GOAL!
But is this optimal? Let’s check if any path can be cheaper.
Since all edge costs positive and heuristic admissible, first time we pop G, it’s optimal.
Final path: A→B→E→G, cost=14.
✅ Done!
💡 Key Insight: Even though C looked promising, heuristic guided us to explore B→E→G which turned out optimal. The heuristic prevented unnecessary expansion of higher-cost branches.

Q.9 Enhance the Min–Max implementation by incorporating Alpha–Beta Pruning.
Solution:
Let’s say we’re evaluating this position (AI to move):
text
X | O | X
---------
O | O | 
---------
X |   | 
AI ('O') can play at (1,2), (2,1), or (2,2).
Without pruning:
Minimax evaluates ALL possible future moves recursively.
With Alpha-Beta:
•	Start evaluating move (1,2) 
o	Suppose it leads to AI win → score = +9
o	Set α = 9 (best so far for Max)
•	Now evaluate move (2,1) 
o	As we explore, suppose we find a path where human can force a win → score ≤ -8
o	Since β (-8) ≤ α (9) → we prune (stop evaluating) this branch!
•	Skip unnecessary computation.
💡 Result: Same optimal move chosen, but much faster!
